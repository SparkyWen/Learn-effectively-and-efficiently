Why two commands? Because OpenClaw has a two-layer sandbox system.
Mode controls when sandboxing kicks in.
Non-main means agents and group chats are sandboxed, but your direct messages (DM) have full access.
Sandbox scope controls the isolation boundary. Agent means each agent gets its own sandbox.

Type these three commands.
First line.

Second line, docker-compose run --rm opencloud-cli config set agents.defaults.sandbox.docker.readonly_root space true.
Third line.
docker-compose run --rm openclaw-cli configset agents.defaults.sandbox.docker.cap_drop '["all"]'
Why three commands? Each one locks down a different attack vector.
Network: none. The sandbox has no internet access. If an attacker tries to download malware or send your data somewhere, it fails.
Read-only root: true. The sandbox's file system is read-only. If an attacker tries to install persistent malware, it fails.
CAP_DROP all drops all Linux capabilities. The sandbox can't do anything privileged. This creates a tradeoff: the benefit, attackers can't download malware or delete system files; the cost, your bot can't install new tools, like Python packages or browsers, inside the sandbox. For a security-focused setup, this is exactly what we want. We manually install tools. The bot just uses them.

Why two commands?
The allow command tells OpenCLAW which tools can run inside the sandbox, executing commands, reading files, writing files, editing, and applying patches.
The deny command then blocks the browser tool specifically.
